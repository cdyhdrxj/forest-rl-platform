# Платформа ForestRobotTwin

Единая платформа для моделирования, обучения, испытаний и анализа интеллектуального поведения агентов в лесной среде.

Проект объединяет несколько взаимосвязанных постановок: прокладку маршрута через труднопроходимый лесной участок, управление движением мобильного робота в сложной природной среде и патрулирование территории с обнаружением событий. Все режимы работают на общей инфраструктуре: симулятор лесной среды, единые контракты данных, единый контур экспериментов и единая визуализация результатов.

## Назначение проекта

Платформа предназначена для совместной разработки и исследования алгоритмов обучения с подкреплением и базовых методов планирования в лесной среде. Она позволяет описывать сценарии, запускать эксперименты, хранить артефакты, сравнивать методы и воспроизводить результаты.

## Основные режимы работы

### Прокладка троп

Поиск маршрута минимальной стоимости через лесной участок с учётом рельефа, препятствий, риска и труднопроходимых зон.

### Управление роботом

Локальная навигация мобильного робота по сенсорным данным, где политика получает наблюдения из симулятора и формирует команды движения.
Для этого режима архитектура разделена на четыре роли: `Environment Adapter` для обучения и оценки, `Policy Runner` для исполнения модели, `Safety Supervisor` для фильтрации опасных команд и `Robot Bridge` для работы с ROS 2.

### Патрулирование

Регулярный обход территории с максимизацией покрытия и эффективности обнаружения событий, таких как пожары и нарушения.

### Многоагентный режим

Моделирование взаимодействия нескольких агентов в лесной среде, где они обучаются координировать действия, распределять задачи и совместно достигать целей, таких как патрулирование территории или обнаружение пожаров.

## Архитектура проекта

Система состоит из симулятора лесной среды, вычислительного контура обучения и оценки, диспетчера экспериментов, хранилища результатов и веб-интерфейса.

Симулятор отвечает за рельеф, препятствия, события, датчики и исполняемую модель среды.

Вычислительный контур включает алгоритмы обучения с подкреплением, классические базовые методы и модули оценки качества.

Диспетчер экспериментов управляет сценариями, запусками, метриками, журналами эпизодов и артефактами моделей.

Веб-интерфейс предоставляет средства визуализации, сравнения запусков, просмотра реплеев и управления экспериментами.

## Контракты проекта

Общие контракты системы хранятся в каталоге `contracts/`. Это центральная точка согласования между всеми модулями.

`contracts/v1/scenario.schema.json` задаёт формат сохранённого `scenario.json`.

`contracts/v1/preview.schema.json` задаёт формат сохранённого `preview.json`.

`contracts/v1/episode_log.schema.json` задаёт канонический формат экспорта журнала эпизодов.

`contracts/v1/replay.schema.json` задаёт формат одной строки `replay_*.jsonl`.

`contracts/v1/metrics.schema.json` задаёт канонический формат экспорта сводных и временных метрик.

`contracts/openapi.yaml` задаёт HTTP-метаданные backend-сервиса.

`contracts/websocket_protocol.md` фиксирует runtime WebSocket-протокол диспетчера экспериментов.

`contracts/v1/ros_interfaces.md` и `contracts/v2/ros_interfaces.md` фиксируют версии соглашений по топикам, сервисам и сообщениям ROS 2.

Все изменения контрактов должны быть согласованы до внесения изменений в прикладные модули.

## Распределение по студентам

### Яна. Прикладная математика и информатика — прокладка троп

Основная зона ответственности: `services/trail_planning`, `packages/baselines`, `experiments/benchmark_suites`.

### Владимир. Информационные системы и технологии — симулятор лесной среды

Основная зона ответственности: `apps/simulator`, `services/scenario_generator`, `ros2_ws/src/forest_sim_gazebo`.

### Алина. Информационные системы и технологии — веб-система и API

Основная зона ответственности: `apps/api`, `apps/web`, `packages/db`, `contracts/openapi.yaml`.

### Елизавета. Программная инженерия — управление роботом и интеграция

Основная зона ответственности: `services/robot_control`, `packages/env_bridge`, `ros2_ws/src/robot_adapter`, `tests/e2e`, `deploy`, `.github/workflows`.

### Георгий. Прикладная математика и информатика — патрулирование

Основная зона ответственности: `services/patrol_planning`, `packages/rl_algorithms`, `experiments/configs`, `experiments/benchmark_suites`.

### Дмитрий. Программная инженерия — многоагентный режим

Основная зона ответственности: `services/marl_coordination`, `experiments/configs/marl/`, `docs/experiments/marl/`.

## Логика каталогов

`apps/` содержит исполняемые приложения.

`docs/` содержит документацию проекта.

`services/` содержит предметную логику по основным задачам проекта.

`packages/` содержит общие библиотеки.

`ros2_ws/` содержит рабочее пространство ROS 2.

`experiments/` содержит конфигурации запусков, ноутбуки и наборы сценариев.

`tests/` содержит модульные, интеграционные и сквозные тесты.

`deploy/` содержит материалы для развёртывания.

## Принципы разработки

Все общие схемы и соглашения хранятся только в `contracts/`.

Все эксперименты должны быть воспроизводимыми: сценарий, seed, параметры алгоритма и артефакты должны сохраняться.

Любое изменение интерфейсов должно сопровождаться обновлением тестов и документации.

## Запуск проекта

Все команды выполняются из корня проекта.

**Первый запуск** (или после изменения зависимостей `packages/common/requirements.txt`):

```
docker compose up --build
```

> **Важно:** Первая сборка может занять **5–10 минут**.

**Обычный запуск** уже собранного проекта:

```
docker compose up -d
```

## Работа с базой данных

Компоненты, связанные с базой данных, размещены в каталоге `packages/`.

* ORM-модели (SQLAlchemy) находятся в `packages/db/models/`.
* Модели Pydantic, предназначенные для API (входные и выходные данные), находятся в `packages/schemas/`.
* Создание и получение сессии БД выполняется через `packages/db/session.py`.
* Примеры работы с БД: `packages/db/examples/`.
