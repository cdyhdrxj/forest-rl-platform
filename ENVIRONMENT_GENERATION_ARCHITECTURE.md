# Генерация среды: текущее состояние и целевая архитектура

## Зачем нужен этот файл

Файл фиксирует договоренность по сценарию генерации среды до и во время внедрения изменений в проект.
Он нужен как опорная точка для дальнейшей доработки `services/scenario_generator`, API-диспетчера, симулятора и обучающих модулей.

## Как есть сейчас

### Общая картина

Сейчас в проекте уже есть:

- диспетчер экспериментов на FastAPI;
- несколько task-specific сервисов обучения;
- 2D-клеточные и 2D-непрерывные среды;
- задел под симулятор и ROS 2 контракты;
- модели БД для сценариев, их версий и слоев.

При этом отдельный модуль генерации среды пока не оформлен как полноценный компонент.
Каталог `services/scenario_generator` существует, но по сути является заглушкой.

### Где реально происходит генерация сейчас

Генерация среды распределена по нескольким местам:

1. В API-сервисах параметры с фронтенда напрямую передаются в task-specific сервисы.
2. Task-specific сервисы сами собирают объект среды из параметров.
3. Внутри самих сред часть мира генерируется на `reset()`.

Это видно на примерах:

- `services/patrol_planning/service/service.py` создает `GridWorld` из `GridWorldConfig`;
- `services/reforestation_planting/service.py` создает `SeedlingPlantingEnv` из `PlantingEnvConfig`;
- `services/trail_camar/service.py` напрямую создает `CamarGymWrapper`;
- `services/patrol_planning/assets/envs/environment.py` сбрасывает и заново наполняет клеточную среду;
- `services/reforestation_planting/environment.py` генерирует layout при каждом `reset()`.

### Что это означает архитектурно

Сейчас проект работает по модели:

- у каждой задачи свой способ построения среды;
- общего контракта генерации почти нет;
- генерация смешана с обучением и исполнением;
- результат генерации не оформлен как отдельная каноническая сущность;
- визуализация живет через текущее состояние training-service, а не через отдельную модель сценария.

### Что уже подготовлено в проекте

Несмотря на это, в проекте уже есть хорошая база для перехода к более строгой архитектуре:

- `scenario_version` хранит seed, конфиги и ссылки на world/preview;
- `scenario_layer` хранит отдельные слои среды;
- `run` уже отделен от `scenario_version`;
- в `contracts/v1/ros_interfaces.md` уже выделены runtime-сервисы и события среды;
- в архитектурной диаграмме уже есть естественное место для отдельной генерации между диспетчером и симулятором.

## Целевая архитектура

### Главный принцип

Генератор среды должен стать отдельным модулем, который:

- принимает единый запрос на генерацию;
- выбирает реализацию под вид среды;
- применяет task-specific расширения;
- валидирует результат;
- возвращает каноническое описание сгенерированной среды.

Важно:
генератор не должен быть частью симулятора и не должен быть размазан по обучающим модулям.

### Что считается результатом генерации

Результат генерации должен быть не объектом runtime-среды, а отдельной сущностью `GeneratedScenario`.

Она должна содержать:

- тип среды;
- тип задачи;
- seed;
- версию генератора;
- нормализованные effective parameters;
- слои среды;
- превью для веба;
- ссылки на артефакты;
- отчет валидации.

Именно эта сущность должна:

- сохраняться в `scenario_version` и `scenario_layers`;
- передаваться в симулятор на загрузку;
- использоваться для предпросмотра в вебе;
- связываться с `run`.

### Разделение ответственности

#### Веб-интерфейс

Отвечает за:

- выбор параметров генерации;
- запуск генерации;
- отображение preview;
- live-визуализацию уже исполняемой среды.

#### Диспетчер экспериментов

Отвечает за:

- прием и первичную валидацию запроса;
- вызов генератора;
- сохранение версии сценария;
- создание `run`;
- передачу команды загрузки сценария симулятору;
- ретрансляцию runtime-обновлений в веб.

#### Генератор среды

Отвечает за:

- общую orchestration-логику генерации;
- выбор family-generator;
- применение task overlay plugins;
- валидацию мира;
- построение preview;
- сериализацию результата.

#### Симулятор

Отвечает за:

- загрузку уже сгенерированного сценария;
- исполнение динамики среды;
- датчики, события и runtime-состояние;
- публикацию изменений во время эпизода.

#### Обучающие модули

Не должны владеть генерацией мира целиком.

Они должны предоставлять только:

- task-specific параметры;
- task-specific overlay logic;
- task-specific validator rules;
- task-specific интерпретацию событий и метрик.

## Целевой сценарий генерации среды

1. Пользователь в вебе выбирает:
   - вид среды;
   - seed;
   - параметры леса;
   - task-specific параметры.
2. Веб отправляет `GenerateEnvironmentRequest` в диспетчер.
3. Диспетчер валидирует общий формат и вызывает `EnvironmentGenerationService`.
4. Генератор выбирает реализацию по `environment_kind`:
   - `grid`
   - `continuous_2d`
   - `simulator_3d`
5. Генератор строит базовые слои леса.
6. Генератор применяет task-specific overlay.
7. Генератор валидирует результат.
8. Генератор возвращает `GeneratedScenario`.
9. Диспетчер сохраняет `scenario_version` и `scenario_layers`.
10. Диспетчер создает `run`.
11. Диспетчер передает симулятору команду загрузки сценария.
12. Симулятор загружает среду и начинает runtime-исполнение.
13. Для клеточной и 2D-непрерывной среды initial snapshot уходит в веб сразу.
14. Во время эпизода обновления публикует уже симулятор, а диспетчер пересылает их в веб.

## Как должны быть устроены генераторы

### Общий фасад

Нужен один общий entrypoint:

- `EnvironmentGenerationService.generate(request)`

Это общий механизм не в смысле одного универсального алгоритма, а в смысле одного контракта и одного pipeline.

### Family generators

Под каждый вид среды должна быть своя реализация:

- `GridFamilyGenerator`
- `Continuous2DFamilyGenerator`
- `Simulator3DFamilyGenerator`

Они создают базовую геометрию и слои мира.

### Task overlays

Поверх family-generator должны накладываться task-specific плагины:

- патрулирование;
- пожары;
- нарушители;
- посадка леса;
- прокладка маршрута.

Такой подход дает:

- единый pipeline;
- разные реализации под разные среды;
- расширяемость без дублирования генерации в task-service.

## Передача данных между модулями

### Диспетчер -> Генератор

На текущем этапе достаточно обычного вызова application service внутри backend.
Отдельный REST или ROS 2 для этого не нужны.

### Генератор -> Симулятор

Передавать лучше не большой runtime-объект, а идентификатор и артефакты сценария:

- `scenario_version_id`
- `world_file_uri`
- `layers`

То есть фактически:

- генератор формирует сценарий;
- диспетчер сохраняет его;
- симулятор загружает по ссылке/идентификатору.

### Генератор/Диспетчер -> Веб

Для initial preview:

- обычный HTTP-ответ или initial WebSocket message.

Для изменений в процессе исполнения:

- WebSocket.

### Runtime-связь

Если среда изменяется во время эпизода, главным владельцем актуального состояния становится не генератор, а симулятор.
Генератор участвует только в создании initial state и правил.

## Что нужно сохранять в хранилище

Сохранять только seed и параметры недостаточно.

Нужно сохранять:

- исходный запрос на генерацию;
- effective parameters после нормализации;
- seed;
- generator version;
- validation report;
- preview;
- world file;
- слои среды;
- связь с `scenario_version` и `run`.

Это нужно для воспроизводимости даже после изменения логики генератора.

## Параллельные эксперименты

Поддержку нескольких одновременных экспериментов нужно закладывать сразу.

Сейчас в API используются singleton-service экземпляры на маршрут, поэтому параллельность по одной и той же среде ограничена.
Целевая модель должна быть `run`-scoped:

- отдельный `run_id`;
- отдельное состояние;
- отдельный канал обновлений;
- при необходимости отдельный namespace/процесс симулятора.

## Валидация

Валидация должна быть многоуровневой:

1. Диспетчер:
   - схема запроса;
   - диапазоны параметров;
   - совместимость полей.
2. Генератор:
   - корректность построенного мира;
   - наличие обязательных объектов;
   - геометрические и логические ограничения.
3. Task plugin:
   - правила конкретной задачи.
4. Симулятор:
   - можно ли загрузить и исполнить сценарий.

## Что начинаем менять в коде

Первый этап внедрения:

1. Оформить `services/scenario_generator` как реальный модуль.
2. Вынести общие модели запроса и результата генерации.
3. Ввести registry family-generators и task-overlays.
4. Подвязать существующие 2D среды к новому генерационному слою без поломки текущего API.
5. Подготовить почву для последующей интеграции:
   - с `scenario_version`;
   - с `scenario_layers`;
   - с загрузкой сценария в симулятор.

На этом этапе цель не переписать весь проект за один шаг, а перевести генерацию из разрозненной логики в управляемую архитектуру с ясными точками расширения.
